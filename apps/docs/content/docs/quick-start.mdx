## Prerequisites

- Docker & Docker Compose
- A public domain (e.g., `your.domain`)
- Wildcard DNS or subdomain configuration support

## One-Click Start

```bash
# Clone the repository
git clone https://github.com/ZingerLittleBee/EXLO
cd EXLO

# Copy environment variable template
cp .env.docker.example .env

# Edit configuration
# Set POSTGRES_*, BETTER_AUTH_SECRET, INTERNAL_API_SECRET, and API_BASE_URL
# (If INTERNAL_API_SECRET or API_BASE_URL are missing, add them to .env)

# Start services
docker compose up -d
```

## First-Time Setup

If this is a fresh install, create the initial admin account first:

```
http://your.domain/setup
```

## First Connection

### 1. Create SSH Tunnel

```bash
# Random subdomain
ssh -R 8000:localhost:8000 -p 2222 .@your.domain

# Custom subdomain (e.g., myapp.your.domain)
ssh -R 8000:localhost:8000 -p 2222 myapp@your.domain
```

Parameter explanation:
- `-R 8000:localhost:8000`: Forward local port 8000
- `-p 2222`: SSH server port
- Username determines the subdomain:
  - `.` = random subdomain (auto-generated)
  - `myapp` = custom subdomain (`myapp.your.domain`)
  - If not specified, SSH uses your system username (e.g., `ssh -R 8000:localhost:8000 -p 2222 your.domain` uses your local username as the subdomain)

> [!IMPORTANT]
> **Port Consistency Requirement**
> 
> The remote port (first number) and local port (second number) **must be identical** in the `-R` parameter. Due to SSH protocol limitations, the server cannot determine your actual local port, so it relies on the remote port you specify to match your local service port.
>
> **Note**: Mismatched ports will **NOT** cause the connection to close immediately, but the tips displayed in the terminal may be incorrect. Always ensure the ports match.

## Disconnect

Double-press `ESC` in the tunnel terminal to disconnect.

You can also press `Ctrl+C` to close the SSH tunnel. If you're in a nested SSH session or it doesn't exit, use the escape sequence:

```
Press Enter, then type ~.
```

## Reconnection

EXLO supports automatic session recovery:

- **Reconnect within 30 minutes**: No need to re-authenticate via Device Flow
- **Same subdomain preserved**: Your tunnel URL remains unchanged after reconnection
- **Works with both custom and random subdomains**: When using `.` (random), reconnecting within 30 minutes with the same SSH key and port will restore your previous subdomain

> [!NOTE]
> Reconnection is based on SSH public key fingerprint. If you switch devices or SSH keys, re-authentication is required.

## Custom Subdomain Behavior

| Scenario | Subdomain |
|----------|-----------|
| `ssh ... myapp@host` | Uses `myapp`, disconnects on conflict |
| `ssh ... myapp@host` (reconnect with same key+port) | Restores `myapp` |
| `ssh ... .@host` (first time) | Random generated |
| `ssh ... .@host` (reconnect within 30 min) | **Restores previous subdomain** |
| `ssh ... .@host` (after 30 min) | New random subdomain |

## Auto-Reconnect with autossh

Use `autossh` for automatic reconnection on network failures:

```bash
# Install autossh
# macOS
brew install autossh
# Ubuntu/Debian
apt install autossh

# Connect with autossh
autossh -M 0 \
  -o "ServerAliveInterval=30" \
  -o "ServerAliveCountMax=3" \
  -o "ExitOnForwardFailure=yes" \
  -R 80:localhost:3000 -p 2222 user@your.domain
```

Parameter explanation:
- `-M 0`: Use SSH's built-in keepalive (ServerAlive)
- `ServerAliveInterval=30`: Send heartbeat every 30 seconds
- `ServerAliveCountMax=3`: Reconnect after 3 consecutive failures
- `ExitOnForwardFailure=yes`: Exit on forward failure (allows autossh to retry)
